# 哈希缓存方案设计

## Context

当前 fuscan 的扫描结果无法跨次复用：`Scanner.scan()` 每次都对所有文件重新提取内容并应用规则；
`IncrementalScanner` 仅基于 `mtime` 跳过未变化文件，且**不存储扫描结果**——下次想看报告还得重扫。
用户需求是建立"路径无关、规则感知、结果可复用"的缓存：

1. 规则文件解析为单条规则并计算规则哈希；一个规则文件名 → (文件哈希, 多个规则名+规则哈希)
2. 被扫描文件以内容哈希为主键（与路径解耦）
3. 缓存键为 `(file_content_hash, rule_hash)`，值为完整 `RuleHit`
4. 下次扫描：文件哈希未变 + 规则哈希未变 → 直接复用结果，跳过匹配

经对比，新方案在路径变更、规则变更、结果复用三个维度上严格优于原 mtime 方案。
用户已确认按"全链路接入 + 替换 IncrementalScanner + SQLite + 维持 96% 覆盖率门槛"执行。

## 总体架构

```

│                       CacheStore (SQLite)                         │

│  scanned_files / file_paths                                      │
│  scan_results (PK: file_hash, rule_hash)                         │
└──────────────────────────────────────────────────────────────────┘
                            ▲
                            │
        ┌───────────────────┼───────────────────┐
        │                   │                   │

```

## 数据库 Schema（SQLite）

```sql
PRAGMA user_version = 1;
PRAGMA journal_mode = WAL;     -- 并发读不阻塞写
PRAGMA foreign_keys = ON;

-- 规则文件：文件名 + 文件哈希
CREATE TABLE IF NOT EXISTS rule_files (
    file_path  TEXT PRIMARY KEY,    -- 规则文件绝对路径
    file_hash  TEXT NOT NULL,       -- 规则文件内容 SHA-256
    mtime      REAL NOT NULL,
    loaded_at  TEXT NOT NULL
);

-- 规则条目：规则哈希唯一标识（跨文件去重，相同规则只存一份）
CREATE TABLE IF NOT EXISTS rules (
    rule_hash  TEXT PRIMARY KEY,
    rule_name  TEXT NOT NULL,
    severity   TEXT,
    description TEXT,
    serialized TEXT NOT NULL        -- 规则的稳定 JSON 序列化（便于审计）
);

-- 规则文件与规则的多对多（一个规则文件含多条规则；同一规则可被多文件载入）
CREATE TABLE IF NOT EXISTS rule_file_members (
    file_path  TEXT NOT NULL,
    rule_hash  TEXT NOT NULL,
    PRIMARY KEY (file_path, rule_hash),
    FOREIGN KEY (file_path) REFERENCES rule_files(file_path) ON DELETE CASCADE,
    FOREIGN KEY (rule_hash) REFERENCES rules(rule_hash) ON DELETE CASCADE
);

-- 被扫描文件：内容哈希主键，与路径解耦
CREATE TABLE IF NOT EXISTS scanned_files (
    file_hash       TEXT PRIMARY KEY,
    size            INTEGER NOT NULL,
    first_seen_at   TEXT NOT NULL,
    last_scanned_at TEXT NOT NULL
);

-- 文件路径索引（同一哈希可对应多个路径，记录最近一次见到的 mtime）
CREATE TABLE IF NOT EXISTS file_paths (
    file_hash    TEXT NOT NULL,
    path         TEXT NOT NULL,
    mtime        REAL NOT NULL,
    last_seen_at TEXT NOT NULL,
    PRIMARY KEY (file_hash, path),
    FOREIGN KEY (file_hash) REFERENCES scanned_files(file_hash) ON DELETE CASCADE
);
CREATE INDEX IF NOT EXISTS idx_paths_path ON file_paths(path);

-- 缓存结果：(file_hash, rule_hash) → 完整 RuleHit
CREATE TABLE IF NOT EXISTS scan_results (
    file_hash   TEXT NOT NULL,
    rule_hash   TEXT NOT NULL,
    matched     INTEGER NOT NULL,        -- 0/1 是否命中
    severity    TEXT,                    -- 命中时填，否则 NULL
    detail      TEXT,
    match_text  TEXT,
    match_count INTEGER NOT NULL DEFAULT 1,
    target      TEXT,
    cached_at   TEXT NOT NULL,
    PRIMARY KEY (file_hash, rule_hash),
    FOREIGN KEY (file_hash) REFERENCES scanned_files(file_hash) ON DELETE CASCADE,
    FOREIGN KEY (rule_hash) REFERENCES rules(rule_hash) ON DELETE CASCADE
);
CREATE INDEX IF NOT EXISTS idx_results_file ON scan_results(file_hash);
CREATE INDEX IF NOT EXISTS idx_results_rule ON scan_results(rule_hash);
```

## 核心 API 设计

### `compute_rule_hash(rule: Rule) -> str`

```python
def serialize_rule(rule: Rule) -> str:
    """规则转稳定 JSON 字符串（键排序）。"""
    return json.dumps({
        "name": rule.name,
        "match": _serialize_match(rule.match),
        "description": rule.description,
        "severity": rule.severity.value,
        "file_extensions": list(rule.file_extensions),
    }, sort_keys=True, ensure_ascii=False)
```

`_serialize_match` 递归处理 `LeafMatch`/`AndMatch`/`OrMatch`/`NotMatch`，
枚举用 `.value` 字符串，保证不同 Python 版本下哈希稳定。

### `CacheStore`

```python
class CacheStore:
    def __init__(self, db_path: Path) -> None: ...

    def register_ruleset(
        self, ruleset: RuleSet, source_files: Mapping[Path, str] | None = None
    ) -> dict[str, str]:
        """登记规则集：算文件哈希+规则哈希，写入 rule_files/rules/rule_file_members。
        返回 rule_name -> rule_hash 映射供 Scanner 复用。
        """

    def get_rule_hashes(self) -> dict[str, str]:
        """查询当前已登记的 rule_name -> rule_hash。"""

    def get_cached_hits(
        self, file_hash: str, rule_hashes: Collection[str]
    ) -> dict[str, RuleHit | None]:
        """批量查缓存：返回 rule_hash -> RuleHit（命中）或 -> None（规则未命中已缓存）。
        不在返回字典中的 rule_hash 表示未缓存，需扫描。"""

    def put_result(
        self, file_hash: str, size: int, path: Path, mtime: float,
        rule_hash: str, hit: RuleHit | None, rule_name: str, severity: Severity,
    ) -> None:
        """写入单条结果；hit=None 表示该规则对该文件未命中。"""

    def register_file(self, file_hash: str, size: int) -> None:
        """登记/更新 scanned_files.last_scanned_at。"""

    def register_path(self, file_hash: str, path: Path, mtime: float) -> None:
        """登记/更新 file_paths。"""

    def prune_orphan_rules(self, active_rule_hashes: Collection[str]) -> int:
        """清理不在当前规则集中的旧规则及其缓存。返回清理条数。"""

    def prune_stale_files(self, max_age_days: int = 30) -> int:
        """清理长期未见的 scanned_files 及其缓存。"""

    def stats(self) -> CacheStats:
        """返回缓存统计（文件数/规则数/结果数/占用字节）。"""

    def close(self) -> None: ...
```

### `Scanner` 集成

```python
class Scanner:
    def __init__(
        self,
        ruleset: RuleSet,
        ...,
        cache: CacheStore | None = None,
        source_files: Mapping[Path, str] | None = None,
    ) -> None:
        ...
        self._cache = cache
        if cache is not None:
            cache.register_ruleset(ruleset, source_files)
            self._rule_hashes = cache.get_rule_hashes()  # name -> hash
            self._compiled_with_hash = [
                (rule, matcher, self._rule_hashes[rule.name])
                for rule, matcher in self._compiled
            ]

    def _scan_entry(self, entry: FileEntry) -> ScanResult:
        if self._cache is None:
            return self._scan_entry_uncached(entry)
        return self._scan_entry_cached(entry)
```

`_scan_entry_cached` 流程：
1. 计算 `file_hash = compute_file_hash(entry.path)`（普通文件读字节）
2. 查 `get_cached_hits(file_hash, [rule_hashes])`
3. 命中的规则直接构造 `RuleHit`
4. 未命中的规则执行 `matcher.matches(context)`，调 `put_result` 写入缓存
5. 调 `register_file` / `register_path` 更新元数据

### `default_extract_content_with_hash`

```python
def default_extract_content_with_hash(entry: FileEntry) -> tuple[str, str]:
    """默认带哈希的内容提取：读字节 → 算 SHA-256 → 调 extract_content。

    复用同一份 I/O，仅多一次哈希计算。
    """
    if entry.is_dir or entry.size > 50 * 1024 * 1024:
        return "", _hash_bytes(b"")
    data = entry.path.read_bytes()
    return extract_content(entry.path), hashlib.sha256(data).hexdigest()
```

### `IncrementalScanner` 替换

```python
class IncrementalScanner:
    def __init__(
        self, ruleset, cache: CacheStore, ..., source_files=None,
    ) -> None:
        self._scanner = Scanner(ruleset, cache=cache, ...)

    def scan(self, root: Path) -> ScanReport:
        return self._scanner.scan(root)

    def scan_paths(self, paths: list[Path]) -> ScanReport:
        """扫描指定路径列表：逐个调 scan_file，合并结果。"""
        results = [self._scanner.scan_file(p) for p in paths if p.exists() and not p.is_dir()]
        return ScanReport(root=Path(), results=tuple(results), stats=_merge_stats(results))

    # save_state/load_state 改为操作 CacheStore（已自动持久化），保留空实现兼容
```

`save_state`/`load_state` 保留方法但内部空操作（CacheStore 已实时持久化），并加 `DeprecationWarning`。

### 压缩包内条目缓存

`ArchiveScanner._scan_entry` 的 content_provider 闭包改为：

```python
def hashing_content_provider(_fe: FileEntry) -> tuple[str, str]:
    data = reader.read_entry(entry.entry_name)
    return self._decode_or_extract(data, entry), hashlib.sha256(data).hexdigest()
```

`ArchiveScanner.__init__` 接受可选 `cache: CacheStore | None`，逻辑同 `Scanner`。
压缩包内条目的"path"用 `entry.display_path`（如 `zip.zip!inner/file.txt`），
缓存键是内容哈希——路径无关，符合要求。

## 关键决策与依据

### 1. 哈希算法选 SHA-256

- 碰撞概率可忽略（2^-128）
- Python `hashlib` 标准库自带，无新依赖
- 性能约 500 MB/s，对 50MB 上限的文件开销可接受
- 不用 xxhash 等第三方库，避免新依赖审批

### 2. SQLite 单连接 + RLock

- `check_same_thread=False` + 模块级 `threading.RLock` 序列化所有写操作
- 读操作（`get_cached_hits`）也走锁，避免读到半写状态
- WAL 模式提升并发读性能
- 不用连接池：Scanner 主要在 `_scan_entry` 中频繁小查询，单连接 + 预编译 statement 最优

### 3. 规则哈希基于稳定 JSON 序列化

- 不用 `pickle`（跨版本不稳定且有安全风险）
- 不用 `repr(rule)`（dataclass repr 含类型名，Python 版本敏感）
- 自定义 `_serialize_match` 递归处理，枚举用 `.value`，键排序，UTF-8 编码

### 4. 缓存键粒度 `(file_hash, rule_hash)` 单条级

- 用户需求明确："再看规则哈希是否变化"
- 单条级缓存支持"规则部分变更只重扫变更规则"
- 整文件级缓存（`file_hash -> ScanResult`）会因任一规则变化导致整文件重扫

### 5. `IncrementalScanner` 保留为薄包装

- 不直接删除类（破坏 watcher/tray 调用方）
- 内部委托给带 cache 的 `Scanner`
- `scan_paths` 兼容签名保留
- `save_state`/`load_state` 改为 no-op + DeprecationWarning

### 6. 默认缓存路径 `~/.fuscan/cache.db`

- 与 `config.yaml` 同目录，集中管理
- 用户可通过 `--cache-path` 或 Config 自定义

### 7. ArchiveScanner 的 content_provider 闭包返回 `(content, hash)`

- 哈希基于条目原始字节（解压后），而非压缩包整体
- 同一文件打入不同压缩包，缓存仍可复用
- 压缩包密码变化 → 解压后字节可能不同 → file_hash 变化 → 触发重扫，符合预期
